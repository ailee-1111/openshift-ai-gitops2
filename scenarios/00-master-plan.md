## 1. 시연 철학

> "기능 목록을 나열하지 말고, 고객의 하루를 보여줘라."

각 시나리오는 **상황(Context) → 문제(Problem) → 해결(Solution) → 확인(Verification) → 어필(Takeaway)** 5단계 스토리로 구성한다. 시연자는 역할(운영자/개발자/관리자/인프라)을 명시하고, 해당 역할의 권한으로 실제 조작한다.

## 2. 역할 정의

| 역할 | 약어 | OCP 계정 | RBAC | 주요 관심사 |
|------|------|---------|------|-----------|
| 인프라 담당자 | INFRA | poc-admin | cluster-admin | 클러스터 안정성, GPU 활용률, 비용 |
| 개발팀 관리자 | MGR | poc-operator | NS edit + approval | 모델 배포 승인, 팀 리소스 할당, SLA |
| 데이터 사이언티스트 | DS | poc-user | NS view + serving | 모델 등록/서빙, 파이프라인, 평가 |
| 운영자 | OPS | poc-operator | NS edit + monitoring | 모니터링, 알림, 장애 대응, 감사 |

## 4. 시나리오 파일 구조 (각 MD 공통)

```
# S{N}: {시나리오명}

## 메타 정보
(주역할/보조역할/데모시간/검증항목/구축런북/검증런북/IaC)

## 상황 (Context)
> 이런 상황이 발생했습니다... (현대모비스 맥락)

## 문제 (Problem)
> 기존 방식에서는 이런 문제가 있습니다...

## 해결 (Solution) — RHOAI로 이렇게 해결합니다
### Step 1~N: 시연 단계
- 누가 (역할 + 계정)
- 무엇을 (수행 내용)
- 어떻게 (구체적 명령어/UI 조작)
- 권한 (RBAC)
- 확인 (검증 방법)

## 확인 (Verification)
| 검증 기준 | 기대값 | 실측값 |

## 이번 시연에서 확인된 핵심 가치
> 이 시연에서 입증된 것

## 추천 사항
> 이 기능은 이렇게 사용하면 좋습니다
```
